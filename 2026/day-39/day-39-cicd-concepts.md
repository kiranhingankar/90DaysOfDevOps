# Day 39 – CI/CD Concepts

## What is CI/CD?

CI/CD is a modern software development practice that helps teams build, test, and deploy applications automatically and reliably.

Instead of manually testing and deploying code, CI/CD pipelines automate the process so teams can release updates faster with fewer errors.

---

# Task 1 – The Problem

## 1. What can go wrong in manual deployments?

When multiple developers push code and deploy manually, many problems can happen:

- Code conflicts between developers
- Deployment steps may be forgotten
- Bugs can reach production
- Different environments may behave differently
- Rollbacks become difficult
- Human errors increase
- Downtime may occur during deployment
- One broken feature can affect the whole application

---

## 2. What does “It works on my machine” mean?

“It works on my machine” means the application works correctly on a developer’s local system but fails on another machine or server.

This happens because:
- Different operating systems
- Missing dependencies
- Different environment variables
- Different software versions
- Untracked local configurations

CI/CD solves this by creating consistent automated environments for testing and deployment.

---

## 3. How many times can a team safely deploy manually?

Manual deployments are slow and risky.

Most teams can safely deploy only a few times per day manually because:
- Every deployment requires human verification
- Testing takes time
- Mistakes are common
- Rollbacks are difficult

With CI/CD automation, companies can deploy many times a day safely.

---

# Task 2 – CI vs CD

## 1. Continuous Integration (CI)

Continuous Integration is the practice of automatically building and testing code whenever developers push changes to a shared repository.

It helps detect bugs, integration issues, and broken code early.

### Real-world Example
A developer pushes code to GitHub, and GitHub Actions automatically runs tests and checks code quality.

---

## 2. Continuous Delivery (CD)

Continuous Delivery means the application is always ready for deployment after passing all automated tests.

The deployment to production still requires manual approval.

### Real-world Example
An e-commerce company automatically prepares a production-ready release, but the manager clicks “Deploy” manually.

---

## 3. Continuous Deployment

Continuous Deployment automatically deploys every successful change directly to production without manual approval.

Only fully tested code reaches users.

### Real-world Example
Netflix or Facebook automatically deploys updates after automated testing passes.

# Task 3 – Pipeline Anatomy

## Trigger
A trigger starts the pipeline.

Examples:
- Git push
- Pull request
- Schedule
- Manual execution

---

## Stage
A stage is a logical phase in the pipeline.

Examples:
- Build
- Test
- Deploy

---

## Job
A job is a group of tasks executed inside a stage.

Example:
- Running unit tests
- Building Docker image

---

## Step
A step is a single command or action inside a job.

Examples:
- `npm install`
- `docker build`
- `pytest`

---

## Runner
A runner is the machine or environment that executes the pipeline jobs.

Examples:
- GitHub-hosted runner
- Self-hosted server

---

## Artifact
An artifact is the output generated by a job.

Examples:
- Docker image
- Build package
- Test reports

---
